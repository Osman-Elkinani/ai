👑 LEADER — Project Overview & Architecture

النظرة الشاملة وهيكلة المشروع

1. فكرة المشروع والهدف الأكاديمي

المشروع هو نظام AI يقدم Personalized Health & Fitness Recommendations. الفكرة الأساسية هي تحويل مشكلة "كيف أحسّن لياقتي؟" إلى مشكلة رياضية قابلة للحل عبر أدوات الـ AI اللي درسناها في المنهج.

1.1 ليش اخترنا Reinforcement Learning وليس Neural Networks؟

الـ Supervised Learning يحتاج Labeled Dataset ضخم من بيانات Users حقيقيين. بينما الـ Reinforcement Learning يتعلم عبر Trial and Error داخل Simulation Environment — وهذا بالضبط ما نملكه: بيئة MDP مبنية على OpenAI Gymnasium تحاكي استجابة الجسم البشري.

Simulation Platform: Python + OpenAI Gymnasium (كما هو مطلوب في الـ Task Description).

2. الربط بالمنهج — Theoretical Foundations

كل مكون في المشروع مربوط مباشرة بمحاضرة محددة. هنا نشرح النظرية أولاً ثم نوضح كيف طبقناها.

2.1 Markov Decision Process — MDP (Lectures 7 & 8)

في الـ Lecture 7 درسنا إن أي Decision-Making Problem يمكن نمذجته كـ MDP لو توفرت 5 عناصر:

العنصرالتعريف من المنهجالتطبيق في مشروعنا
S (States) مجموعة كل الحالات الممكنة للبيئة Tuple من 6 متغيرات: (fitness, energy, weight, muscle, fatigue, day) = 43,750 State
A (Actions) مجموعة الأفعال المتاحة في كل State 28 Action مركب: 7 Workouts × 4 Nutrition Plans
T(s,a,s') Transition Function — احتمالية الانتقال من State لأخرى دالة step(action) في fitness_env.py مع 10% Stochastic Noise
R(s,a,s') Reward Function — إشارة التعزيز دالة _calculate_reward() — Composite Function بـ 4 أوزان (Goal Progress, Energy, Fatigue Penalty, Safety)
γ (Gamma) Discount Factor — أهمية المكافآت المستقبلية γ = 0.95 — عالية لأن اللياقة تتطلب Long-Term Planning

💡 Markov Property

من Lecture 7: الـ MDP يفترض أن الـ Future يعتمد فقط على الـ Current State وليس على الـ History. في مشروعنا، الـ State Tuple (fitness=4, energy=3, weight=2, muscle=1, fatigue=0, day=2) يحتوي كل المعلومات اللازمة لاتخاذ القرار — ما نحتاج نعرف شنو سوى المستخدم أمس.

2.2 Bellman Equation (Lecture 8)

في Lecture 8 درسنا أن القيمة المثلى لأي State تُحسب بمعادلة Bellman:

V*(s) = max_a [ R(s,a,s') + γ · V*(s') ]

هذه المعادلة تقول: "قيمة أي State = أفضل Reward فوري + قيمة الـ Next State مضروبة في γ". طبقنا هذا حرفياً في value_iteration_agent.py — الـ Agent يمر على كل الـ 6,250 States ويطبق هذه المعادلة حتى الـ Convergence.

2.3 Q-Learning — Temporal Difference Learning (Lecture 9)

في Lecture 9 درسنا إن الـ Agent يقدر يتعلم بدون ما يعرف الـ Transition Model (Model-Free). يتعلم فقط من Experience:

Q(s,a) ← Q(s,a) + α [ R + γ · max_a' Q(s',a') - Q(s,a) ]
                         ↑                              ↑
                    TD Target                    Current Estimate
                         └──── TD Error ────────────┘

طبقنا هذا في q_learning_agent.py — الـ Agent يتفاعل مع الـ Environment عبر env.step(action)، يستقبل الـ Reward والـ Next State، ثم يحدّث الـ Q-Table.

2.4 Exploration vs Exploitation — ε-Greedy (Lecture 9-10)

من أهم المفاهيم في الـ RL: الـ Agent لازم يوازن بين:

  • Exploration: تجربة Actions جديدة ما جربها قبل (عشان يكتشف خيارات أفضل)
  • Exploitation: استغلال أفضل Action يعرفه حالياً (عشان يجمع أكثر Reward)

الحل من المنهج: ε-Greedy Policy — بنسبة ε يختار Random Action، وبنسبة (1-ε) يختار Best Action. في مشروعنا، ε يبدأ من 1.0 (Pure Exploration) وينزل تدريجياً لـ 0.05 عبر Multiplicative Decay.

2.5 Approximate Q-Learning — Feature-Based Generalization (Lecture 10)

في Lecture 10 درسنا مشكلة الـ Curse of Dimensionality: لما يكون الـ State Space كبير جداً، الـ Q-Table ما يقدر يغطي كل الـ States. الحل: نستبدل الـ Table بمعادلة خطية:

Q(s,a) ≈ w₁·f₁(s,a) + w₂·f₂(s,a) + ... + wₙ·fₙ(s,a)

بدل ما نحفظ 1.2 مليون Q-Value (43,750 States × 28 Actions)، نتعلم 12 Weight فقط. الميزة: الـ Agent يقدر يعمل Generalization — لو شاف State جديدة ما شافها قبل، يحلل الـ Features بتاعتها ويستنتج Best Action.

2.6 Informed Search — A* Algorithm (Lecture 3)

في Lecture 3 درسنا أن A* يضمن إيجاد الـ Optimal Path لو الـ Heuristic كانت Admissible (ما تبالغ في التقدير):

f(n) = g(n) + h(n)
  g(n) = Actual Cost from Start to n
  h(n) = Heuristic Estimate from n to Goal  ← must be Admissible

في مشروعنا، A* يبني خطة 30 يوم كاملة. الـ Heuristic بتاعتنا Admissible لأنها تفترض Best-Case Scenario (أقصى Fitness Gain بدون أي Fatigue) — هذا دائماً Underestimate للـ True Cost.

2.2 مسار البيانات (Data Flow)

خلني أشرح بالضبط شنو يصير لما المستخدم يضغط "Train AI" في الـ Dashboard:

  1. المستخدم يضغط الزر → الـ JavaScript في dashboard.js يرسل POST /api/train مع عدد الـ Episodes.
  2. الـ Flask في app.py يستقبل الطلب ويشغل q_agent.train(env, num_episodes=50).
  3. داخل الـ train() في q_learning_agent.py:
    • يستدعي env.reset() → البيئة تعيد الحالة الأولية من user_profile.py.
    • يستدعي self.get_action(state) → يختار فعل (Epsilon-Greedy).
    • يستدعي env.step(action) → البيئة تحسب الـ Transition باستخدام بيانات من workouts.py و nutrition.py.
    • يستقبل (next_state, reward, done, truncated, info).
    • يستدعي self.update(state, action, reward, next_state, done) → يحدّث الـ Q-Table.
  4. الـ Flask يرجع الـ Metrics كـ JSON → الـ JavaScript يحدّث الـ Learning Curve Chart والـ Progress Bar.
  5. هذا يتكرر على دفعات (Batches) كل 50 episode حتى يصل للعدد المطلوب.

يحتوي على 3 Dependencies فقط: flask>=3.0.0، numpy>=1.24.0، و gymnasium>=1.0.0 (OpenAI Gym). المشروع لا يحتاج TensorFlow أو PyTorch لأنه لا يستخدم Neural Networks.

3.2 مجلد environment/

📄 fitness_env.py (530+ سطر — أطول ملف)

القلب النابض للمشروع. يحتوي على class FitnessEnv(gym.Env) الذي يرث من gymnasium.Env ويمثل الـ MDP بالكامل. يحدد action_space = Discrete(28) و observation_space = MultiDiscrete([10,5,5,5,5,7]). الـ Interface:

  • reset() → يعيد Initial State.
  • step(action) → ينفذ الـ Transition Function ويرجع (next_state, reward, done, truncated, info) — نفس الـ Gymnasium Signature.
  • get_deterministic_next_state(state, action) → Deterministic version للـ A* Search و Value Iteration.
  • state_distance_to_goal(state) → الـ Heuristic Function للـ A*.

📄 user_profile.py (159 سطر)

يحتوي على class UserProfile مع 3 Presets جاهزة:

  • beginner_weight_loss: أحمد، 28 سنة، 92 كجم، هدفه نزول الوزن.
  • intermediate_muscle: خالد، 24 سنة، 75 كجم، هدفه بناء العضلات.
  • advanced_fitness: عمر، 30 سنة، 68 كجم، هدفه اللياقة العامة.

كل Preset يحدد الـ initial_state والـ target_state اللي يستخدمها الـ MDP.

3.3 مجلد data/

📄 workouts.py (166 سطر)

قاعدة بيانات الـ 7 تمارين. كل تمرين عبارة عن Dictionary فيه:

  • requirements: الحد الأدنى المطلوب (مثلاً HIIT يتطلب fitness >= 4 و energy >= 2).
  • calories_burned: السعرات المحروقة (مبنية على معايير ACSM).

📄 nutrition.py (76 سطر)

قاعدة بيانات الـ 4 خطط غذائية: Caloric Deficit (1600 cal)، Maintenance (2200 cal)، Caloric Surplus (2800 cal)، High Protein. كل خطة لها effects و macros (نسب البروتين/الكربوهيدرات/الدهون).

4. شرح config.py سطر بسطر

هذا الملف هو "العقل المركزي" للمشروع. كل ملف آخر يستورد منه. السبب في وجود هذا الملف هو منع التضارب — لو كان الـ DISCOUNT_FACTOR مكتوب يدوياً في 4 ملفات مختلفة، ممكن واحد يكتب 0.95 والثاني يكتب 0.9 ويصير Bug.

4.1 Environment State Space

FITNESS_LEVELS = 1010 Levels للـ Fitness (0-9). هو المتغير الأكبر نطاقاً لأنه الـ Primary Goal لأغلب المستخدمين.
ENERGY_LEVELS = 55 Levels للـ Energy (0-4). Fast-changing variable — يتأثر مباشرة كل Step.
WEIGHT_STATUSES = 55 Levels للـ Weight (0-4). Slow-changing variable — يستخدم Accumulator Pattern.
MUSCLE_LEVELS = 55 Levels للـ Muscle (0-4). Slow-changing أيضاً — يحتاج ~7 أيام Training لتغيير Level واحد.
FATIGUE_LEVELS = 55 Levels للـ Fatigue (0-4). Level 4 يسبب Penalty -4.0 في الـ Reward (Safety Constraint).
DAYS_PER_WEEK = 7Day of Week (0-6). موجودة في الـ State لأن بعض الـ Users يفضلون Rest أيام معينة.
STATE_SPACE_SIZE = 10 × 5 × 5 × 5 × 5 × 7 = 43,750 States

⚠️ Curse of Dimensionality

لو أضفنا متغير واحد فقط (مثل "الوقت المتاح" بـ 3 قيم)، الفضاء يقفز لـ 131,250 حالة. لو أضفنا متغيرين، يصبح +393,750 حالة. الـ Tabular Q-Learning لن يستطيع تغطية هذا الحجم في 5000 Episode.

الحل المعماري: المتغيرات الإضافية (المعدات، الوقت) تعمل كـ Contextual Filters — ترشّح الـ Actions المتاحة قبل دخول الـ MDP، بدل أن تكون جزء من الـ State.

4.2 Action Space

WORKOUT_TYPES = ['rest', 'walking', ...]7 أنواع تمارين. كل واحد له Effects مختلفة على الـ State Variables.
NUTRITION_PLANS = ['caloric_deficit', ...]4 خطط تغذية. الـ Agent يختار واحدة مع كل تمرين.
NUM_ACTIONS = 7 × 4 = 28الفعل المركب: (workout, nutrition). الـ Agent يختار رقم من 0 إلى 27 ويتم فكه بـ action // 4 و action % 4.

4.3 Q-Learning Hyperparameters

LEARNING_RATE = 0.1α (Alpha) — كم "نثق" بالمعلومة الجديدة. 0.1 يعني 10% من التحديث يأتي من التجربة الجديدة و 90% من القيمة القديمة. قيمة منخفضة لأن البيئة Stochastic.
DISCOUNT_FACTOR = 0.95γ (Gamma) — أهمية المستقبل. 0.95 يعني المكافأة بعد 10 أيام تساوي 0.95^10 = 0.60 من قيمتها الفورية. قيمة عالية لأن اللياقة البدنية تتطلب "تضحية اليوم لنتيجة الشهر".
EPSILON_START = 1.0في البداية، الـ Agent عشوائي 100% (Pure Exploration). لا يعرف شيء عن البيئة.
EPSILON_END = 0.05في النهاية، يبقى 5% عشوائي. لماذا لا ننزلها لـ 0؟ لمنع الـ Policy Collapse — لو واجه حالة نادرة لم يتدرب عليها، هذا الـ 5% يساعده يستكشف حل.
EPSILON_DECAY = 0.998كل Episode يُضرب الـ ε في 0.998. بعد 500 Episode: 1.0 × 0.998^500 ≈ 0.37. بعد 3000: ≈ 0.0025 (يصل للـ Minimum).
NUM_EPISODES = 5000عدد المحاولات. كل Episode = محاكاة 30 يوم كاملة. 5000 × 30 = 150,000 خطوة تدريبية.
MAX_STEPS_PER_EPISODE = 30كل Episode ينتهي بعد 30 Step (يوم). يمكن أن ينتهي أبكر إذا تحقق الـ Goal.

4.4 Approximate Q-Learning

APPROX_LEARNING_RATE = 0.01أقل من الـ Tabular (0.01 بدل 0.1). السبب: تحديث الـ Weights يؤثر على كل الـ States مرة واحدة (لأنها معادلة خطية مشتركة)، فلو كان الـ α عالي ستتذبذب الأوزان بشدة.
NUM_FEATURES = 1212 Feature Function تم تصميمها يدوياً (Feature Engineering). بدل 1.2 مليون خانة في الـ Q-Table، نتعلم 12 رقم فقط.

4.5 A* Search

MAX_SEARCH_DEPTH = 30يخطط لمدة 30 يوم (شهر كامل). سبب الحد: الـ Branching Factor = 28، بدون حد ستنفجر الذاكرة.
SEARCH_BEAM_WIDTH = 2000أقصى عدد Nodes يتم توسيعها. يمنع الـ Memory Overflow. 2000 كافية لإيجاد خطة ممتازة.

5. أسئلة المناقشة المتوقعة (Defense FAQ)

س1: هل البيانات دي Dummy Data ولا Real Data؟

الجواب: البيانات نوعين:

  • بيانات التمارين والتغذية: مبنية على معايير ACSM و CDC الأمريكية. مثال: الـ Jogging يحرق 350 سعرة / 30 دقيقة — هذا الرقم من إرشادات الـ American College of Sports Medicine.
  • بيانات الـ AI: كل Q-Value وكل Reward وكل Policy يتم حسابها فعلياً Real-time أثناء التدريب. لا يوجد أي رقم Hardcoded في النتائج.

س2: ليش 5 متغيرات بس في الـ State؟ النوم والوقت والعمر مو مهمين؟

الجواب: بسبب الـ Curse of Dimensionality. كل متغير إضافي يضرب حجم الـ State Space. الـ 5 متغيرات الحالية تعطي 43,750 State. لو أضفنا "الوقت المتاح" (3 قيم) يصبح 131,250. لو أضفنا "النوم" (3 قيم) يصبح 393,750. الـ Tabular Q-Learning ما يقدر يغطي هذا الحجم.

الحل المعماري: العمر والوقت والمعدات تعمل كـ Contextual Filters — ترشّح الأفعال المتاحة قبل الدخول في الـ MDP. مثال: لو المستخدم عنده 30 دقيقة فقط، الـ Swimming (40 دقيقة) يتم حذفها من الـ Action Space تلقائياً عبر get_available_workout_indices().

س3: الـ Learning Curve فيه تذبذب — هل هذا يعني الـ Agent ما تعلم؟

الجواب: التذبذب متعمد وطبيعي ولا يعني فشل التعلم. مصدره شيئين:

  1. Epsilon = 0.05: الـ Agent لسا يسوي أفعال عشوائية في 5% من الخطوات. هذا يسمى Exploration Noise وهو Standard Practice لمنع الـ Local Optima. لو نزّلنا الـ ε لـ 0 الخط يثبت لكن الـ Agent يعلق في سياسة واحدة ولا يقدر يتكيف.
  2. Stochasticity (10%): البيئة فيها 10% احتمالية لحدوث تأثير عشوائي (إرهاق مفاجئ). هذا يحاكي الواقع — حتى لو سويت أفضل تمرين ممكن تصاب بإرهاق غير متوقع.

الدليل على التعلم: الـ Avg Reward يرتفع من -20 (أول 100 Episode) إلى +42 (آخر 100 Episode). هذا يثبت أن الـ Policy تحسنت بشكل واضح.

س4: شنو الفرق بين الـ 4 Agents في مشروعكم تحديداً؟

الجواب:

س5: الـ Avg Reward = 42. هل هذا رقم كويس ولا ضعيف؟

الجواب: هذا رقم ممتاز. الحد الأقصى النظري للـ Reward في Episode واحد (30 يوم) يتراوح بين 50-60 تقريباً. السبب أن دالة الـ Reward فيها عقوبات إجبارية:

  • عقوبة الزمن: -0.1 × 30 = -3.0 في كل Episode
  • عقوبة الإرهاق: -0.5 إلى -4.0 كلما ارتفع الـ Fatigue
  • عقوبة التكرار: -0.3 لو كرر نفس الفعل
  • الـ Stochasticity: 10% عشوائية تقلل الـ Reward

يعني 42 من أصل ~55 ممكن = ~76% من الـ Optimal. وهذا طبيعي جداً في بيئة Stochastic.

س6: Value Iteration يأخذ 60 ثانية وهو بس 6,250 State — ليش بطيء؟

الجواب: لأن الحساب مركب. في كل Iteration، الخوارزمية تمر على:

  • 6,250 State × 28 Action = 175,000 عملية لكل Iteration
  • كل عملية تستدعي get_deterministic_next_state() + _compute_reward() — كل واحدة فيها 20+ سطر حسابات
  • الخوارزمية تحتاج ~40-50 Iteration للـ Convergence
  • المجموع: 175,000 × 50 = 8.75 مليون عملية حسابية في Pure Python (بدون C Extensions)

س7: كيف ضمنتوا أن الـ Heuristic في A* هي Admissible؟

الجواب: الـ Heuristic تحسب المسافة للهدف بطريقة متفائلة جداً (Optimistic). مثلاً لهدف weight_loss:

distance = (current_weight - target_weight) × 2 + (target_fitness - current_fitness) × 1.5

هذا يفترض أن كل خطوة ستنقص الوزن وتزيد اللياقة بأقصى معدل ممكن، بدون أي Fatigue أو Energy Loss. هذا الافتراض مستحيل عملياً، مما يعني أن الـ Heuristic دائماً تقلل من التكلفة الحقيقية (Underestimate)، وهذا بالضبط شرط الـ Admissibility الرياضي.

س8: شنو الـ Accumulator Pattern وليش كان ضروري؟

الجواب: في الواقع، لو جريت يوم واحد ما بتلاحظ فرق في وزنك. وزنك يحتاج أسابيع من النظام عشان ينزل كيلو واحد. بدون الـ Accumulator Pattern، كل step() ستحسب weight + (-0.05) ثم تعمل round()، والنتيجة ستكون دائماً 0 تغيير لأن 0.05 تُقرّب لـ 0.

الحل: نخزّن الكسور في متغير _acc_weight ونراكمها. لما يوصل التراكم لـ 1.0 أو -1.0، نغيّر المستوى الفعلي في الـ State. هذا خلّى A* Search يقدر يشوف تأثير 20 يوم من الدايت على الوزن، بدلاً من أن يرجع خطة فارغة.

س9: كيف بتضمنوا سلامة المستخدم وما توصوه بتمارين خطيرة؟

الجواب: عبر 3 آليات:

  1. Requirements Check: كل تمرين له متطلبات. الـ HIIT يتطلب fitness >= 4 و energy >= 2. لو المستخدم ما يستوفيها، الـ Environment يمنع التمرين ويجبره على Rest مع عقوبة -2.0.
  2. Progressive Fatigue Penalty: عقوبة الإرهاق متصاعدة:
    • Fatigue = 2: -0.5 (تحذير)
    • Fatigue = 3: -2.0 (خطر)
    • Fatigue = 4: -4.0 (إصابة — أقوى عقوبة في النظام)
  3. Energy Depletion Penalty: لو الطاقة وصلت 0: -2.5. هذا يجبر الـ Agent يتعلم أن الراحة ضرورية.

س10: شنو الـ Potential-Based Reward Shaping (PBRS) وليش استخدمتوه؟

الجواب: بدون PBRS، الـ Agent يحتاج آلاف الـ Episodes عشان يكتشف إن الأفعال اللي تقربه من الهدف هي الأفضل — لأن المكافأة الكبيرة (Goal Achievement Bonus) تأتي فقط في نهاية الـ Episode.

الـ PBRS يعطي الـ Agent "تلميحات" مستمرة (Dense Reward) كل خطوة:

shaping = γ × φ(s') - φ(s)
حيث φ(s) = -distance_to_goal(s)

الميزة الرياضية: Ng et al. (1999) أثبتوا أن هذا الشكل المحدد من الـ Shaping لا يغير الـ Optimal Policy. يعني الـ Agent يتعلم أسرع لكن يوصل لنفس النتيجة النهائية — ما فيه "غش".

وكلاء التعلم المعزز — Q-Learning

1. الربط بالمنهج — Model-Free RL (Lectures 9 & 10)

في Lectures 9 و 10، درسنا الـ Model-Free Reinforcement Learning. عكس الـ Value Iteration اللي يحتاج يعرف كل تفاصيل البيئة (الـ Transition Model والـ Reward Function) מسبقاً، الـ Model-Free Agents تتعلم فقط من التجربة (Experience).

في مشروعنا، בניنا שני Agents يمثلون تطور الـ RL:

  • QLearningAgent: يمثل الـ Tabular Q-Learning (Lecture 9) — يحفظ قيمة كل Action في كل State في جدول.
  • ApproxQLearningAgent: يمثل الـ Approximate Q-Learning (Lecture 10) — يستخدم Features وأوزان (Weights) للتعميم (Generalization) والتغلب على الـ Curse of Dimensionality.

2. Tabular Q-Learning — q_learning_agent.py

الـ Q-Learning هو خوارزمية Off-Policy TD Control. يتعلم Q-Values اللي تمثل الـ Expected Future Reward لكل (State, Action) pair.

2.1 الـ Q-Table والتحدي الهندسي (defaultdict)

نظرياً، الـ Q-Table هو مصفوفة (Array) أبعادها `|S| × |A|`. في مشروعنا:

|S| = 43,750
|A| = 28
Q-Table Size = 1,225,000 Cells

لو أنشأنا NumPy Array بهذا الحجم، سيستهلك ذاكرة كبيرة ومعظمها سيكون أصفار لأن الـ Agent مستحيل يزور كل الـ States في 5000 Episode. الحل: استخدمنا collections.defaultdict:

self.q_table = defaultdict(lambda: np.zeros(self.num_actions))

هذا يطبق Lazy Initialization — الـ State لا تُضاف للجدول إلا لما يزورها الـ Agent لأول مرة. هذا يوفر الذاكرة ويسرع البرنامج.

2.2 Bellman Update Rule (TD Learning)

من Lecture 9، تحديث الـ Q-Value يتم عبر معادلة الـ Temporal Difference (TD):

def update(self, state, action, reward, next_state, done):
    # 1. Get current Q-value
    current_q = self.q_table[state][action]

    # 2. Calculate TD Target
    if done:
        target = reward  # No future rewards if episode is done
    else:
        # max_a' Q(s', a')
        best_next_q = np.max(self.q_table[next_state])
        target = reward + self.gamma * best_next_q

    # 3. Update Q-value
    # Q(s,a) = Q(s,a) + alpha * (target - Q(s,a))
    td_error = target - current_q
    self.q_table[state][action] += self.alpha * td_error

💡 ليش نستخدم alpha (Learning Rate)؟

البيئة بتاعتنا Stochastic (فيها عشوائية 10%). لو alpha = 1.0، الـ Agent سيستبدل القيمة القديمة بالجديدة تماماً، وسيكون حساس جداً للـ Noise. اخترنا alpha = 0.1 عشان الـ Agent ياخذ 10% من التجربة الجديدة و90% من الخبرة السابقة، مما يعطي Stable Learning.

2.3 Exploration vs. Exploitation (ε-Greedy)

عشان نضمن إن الـ Agent يستكشف حلول جديدة وما يعلق في Local Optima، نستخدم Epsilon-Greedy Policy:

def get_action(self, state):
    # Exploration: بنسبة epsilon، اختر Random Action
    if random.random() < self.epsilon:
        return random.choice(self.env.valid_workout_indices)
    
    # Exploitation: بنسبة 1-epsilon، اختر Best Action
    return np.argmax(self.q_table[state])

الـ epsilon يبدأ بـ 1.0 (عشوائية كاملة) وينضرب في epsilon_decay = 0.998 كل Episode. ينزل تدريجياً لحد ما يوصل لـ epsilon_end = 0.05 (5% عشوائية دائمة عشان يظل يستكشف لو تغيرت البيئة).

3. Approximate Q-Learning — approx_q_agent.py

الـ Tabular Q-Learning ممتاز، لكن فيه مشكلة قاتلة درسناها في Lecture 10: لا يستطيع التعميم (No Generalization). لو الـ Agent تعلم أن (fitness=5, weight=2) ممتازة للـ Jogging، وواجه حالة (fitness=5, weight=3)، ما راح يعرف يتصرف لأنها State جديدة تماماً بالنسبة للـ Q-Table.

3.1 Feature Engineering — الانتقال من Table إلى معادلة

بدل ما نحفظ قيم لكل State، بنحسب الـ Q-Value كـ Linear Combination of Features:

Q(s,a) = w₁·f₁(s,a) + w₂·f₂(s,a) + ... + w₁₂·f₁₂(s,a)

الـ Features هي دوال رياضية تعطينا خصائص للـ (State, Action). صممنا 12 Feature تعكس الـ Domain Knowledge:

Featureالوصفالمعادلة التقريبية (Concept)
f0: action_intensity كثافة التمرين مقارنة باللياقة workout.intensity / (fitness + 1)
f1: energy_cost هل التمرين يستهلك طاقة أكثر من المتاح؟ 1.0 if workout.energy_cost > current_energy else 0.0
f2: fatigue_risk خطر الإصابة إذا الإرهاق عالي current_fatigue * workout.intensity
f3: goal_alignment هل التمرين يخدم الهدف المباشر؟ 1.0 if workout_benefits_goal else 0.0

والباقي Features للتغذية (Deficit للوزن، Protein للعضلات) والـ Rest (أهمية الراحة لما الطاقة تكون 0).

3.2 حساب الـ Q-Value (الـ Dot Product)

بدل q_table[state][action]، نحسب الـ Q-Value هكذا:

def get_q_value(self, state, action):
    # 1. استخراج الـ 12 Feature
    features = self.extract_features(state, action)
    
    # 2. حساب Dot Product: w1*f1 + w2*f2 + ...
    return np.dot(self.weights, features)

3.3 Weight Update Rule (Stochastic Gradient Descent)

الآن، إحنا ما نحدّث الـ Q-Value، إحنا نحدّث الأوزان (Weights). من Lecture 10:

def update(self, state, action, reward, next_state, done):
    # 1. Calculate TD Error
    current_q = self.get_q_value(state, action)
    if done:
        target = reward
    else:
        best_next_q = self.get_value(next_state)
        target = reward + self.gamma * best_next_q
    
    difference = target - current_q

    # 2. Update Weights
    # w_i = w_i + alpha * difference * f_i(s,a)
    features = self.extract_features(state, action)
    for i in range(self.num_features):
        self.weights[i] += self.alpha * difference * features[i]

✅ ليش Approximate Q أحسن في الـ Generalization؟

لأن الأوزان مشتركة بين كل الـ States. لو تعلّم الـ Agent إن w_energy_cost لازم يكون سالب (عشان يتجنب التمارين اللي تخلص طاقته)، هذا الوزن بيُطبق على أي State يواجهها مستقبلاً، حتى لو ما زارها أبدًا!

4. Training Metrics — كيف نقيس أداء الـ Agents؟

الـ Agents تحسب Metrics مستمرة أثناء الـ Training عشان نعرضها في الـ Dashboard:

  • Avg Reward (Learning Curve): متوسط الـ Reward لكل 50 Episodes. المفروض يرتفع مع الوقت، مما يثبت أن الـ Agent يتعلم Policies أفضل.
  • State Coverage (للـ Tabular فقط): كم نسبة الـ States اللي زارها الـ Agent من أصل 43,750؟ (عادًة ما تتجاوز 10-15%، وهذا يثبت الحاجة للـ Approximate Q).
  • Policy Stability: كم نسبة الـ Actions اللي ما تغيرت عن الـ Episode السابق. لما تقترب من 100%، يعني الـ Agent وصل للـ Convergence (الاستقرار).

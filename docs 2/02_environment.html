بيئة المحاكاة — Markov Decision Process

1. الربط بالمنهج — ليش بنينا Custom Environment؟

في Lecture 7 درسنا إن أي مشكلة Decision-Making يمكن نمذجتها كـ MDP. المشكلة عندنا: "كيف نعطي المستخدم أفضل Workout + Nutrition Recommendation كل يوم بناءً على حالته الحالية؟"

هذه مشكلة Sequential Decision Making كلاسيكية — القرار بتاع اليوم يأثر على حالة بكرة. وهذا بالضبط ما يحله الـ MDP.

بنينا الـ Environment كـ OpenAI Gymnasium Environment — الـ Class يرث من gymnasium.Env ويحدد:

  • action_space = spaces.Discrete(28) — 28 Action ممكن
  • observation_space = spaces.MultiDiscrete([10,5,5,5,5,7]) — 6 State Variables

3.3 Action Encoding

action = workout_idx × 4 + nutrition_idx

# Decoding:

4. Transition Function — step(action)

من Lecture 7: الـ Transition Function T(s, a, s') تحدد شنو يصير لما الـ Agent يتخذ Action في State معين. في بيئتنا، الـ Transition فيها Stochasticity.

4.1 خطوات الـ step() بالتفصيل

Step 1: Decode الـ Action

workout_idx = action // NUM_NUTRITION    # 0-6
nutrition_idx = action % NUM_NUTRITION    # 0-3
workout_key = WORKOUT_TYPES[workout_idx]  # e.g. 'hiit'
nutrition_key = NUTRITION_PLANS[nutrition_idx]  # e.g. 'maintenance'

Step 2: Requirements Check

كل Workout له requirements. لو المستخدم ما يستوفيها، يتم إجبار Rest مع Penalty -2.0:

# HIIT requires fitness >= 4 AND energy >= 2
for req_key, req_val in workout.get('requirements', {}).items():
    if current_val.get(req_key, 0) < req_val:
        requirements_met = False  # Forced rest

هذا يمنع الـ Agent من اختيار تمارين خطيرة للمبتدئين — Safety Constraint.

Step 3: حساب الـ Deltas مع Stochastic Noise

# الـ Effects الأساسية من الـ Workout + Nutrition Data
delta_fitness = w_effects['fitness'] + random.uniform(-0.1, 0.1)
delta_energy  = w_effects['energy'] + n_effects['energy'] + random.uniform(-0.1, 0.1)
delta_weight  = w_effects['weight'] + n_effects['weight'] + random.uniform(-0.1, 0.1)
delta_muscle  = w_effects['muscle'] + n_effects['muscle'] + random.uniform(-0.1, 0.1)
delta_fatigue = w_effects['fatigue'] + n_effects['fatigue'] + random.uniform(-0.1, 0.1)

⚠️ Stochasticity — ليش أضفنا Noise؟

من Lecture 7: الـ MDP ممكن يكون Deterministic أو Stochastic. في الواقع، نفس التمرين ممكن يعطي نتايج مختلفة حسب النوم، الضغط النفسي، الطقس. الـ random.uniform(-0.1, 0.1) يمثل هذه العشوائية بنسبة ~10%. هذا يخلي الـ Environment أقرب للواقع ويجبر الـ Agent يتعلم Policy أكثر Robust.

Step 4: تحديث الـ Fast-Changing Variables (فوري)

# Energy و Fatigue يتغيرون مباشرة كل Step
new_energy  = clamp(0, 4, round(energy + delta_energy))
new_fatigue = clamp(0, 4, round(fatigue + delta_fatigue))

Step 5: تحديث الـ Slow-Changing Variables (Accumulator Pattern)

هنا أهم Innovation في الـ Environment:

# نراكم الكسور بدل ما نقرّبها فوراً
self._acc_fitness += delta_fitness    # مثال: 0 + 0.15 = 0.15

# لما يوصل التراكم لـ 1.0 → نغير الـ Level الفعلي
if abs(self._acc_fitness) >= 1.0:
    shift = int(self._acc_fitness)    # +1 أو -1
    new_fitness = clamp(0, 9, fitness + shift)
    self._acc_fitness -= shift        # نصفّر المراكم

# نفس الشي للـ weight و muscle

✅ ليش الـ Accumulator Pattern ضروري؟

بدونه: fitness + 0.15 → round() = 0 → ما يتغير شي أبداً!

معاه: بعد 7 أيام Jogging: 0.15 × 7 = 1.05 → shift = +1 → Fitness Level يرتفع!

هذا يحاكي الواقع — اللياقة ما تتحسن بيوم، تحتاج أسابيع من الالتزام.

Step 6: Day Increment

new_day = (day + 1) % 7   # Cyclic: 6 (Sunday) → 0 (Monday)

5. Reward Function — _calculate_reward()

من Lecture 7: الـ Reward Function R(s, a, s') هي "الإشارة" اللي يتعلم منها الـ Agent. تصميمها هو أهم قرار هندسي في أي RL System — لو كانت غلط، الـ Agent يتعلم Policy خاطئة.

5.1 Composite Reward Structure

الـ Reward عندنا مركب من عدة Components بأوزان مختلفة حسب الـ Goal:

R_total = W₁(Goal Progress) + W₂(Energy Maintenance)
        - W₃(Fatigue Penalty) - W₄(Safety Penalty)
        + Diversity Bonus - Time Penalty + PBRS

5.2 Goal-Specific Rewards (W₁)

كل Goal Type له Reward مختلف:

GoalPrimary RewardSecondary RewardBonus
weight_loss (old_w - new_w) × 3.0 (new_f - old_f) × 1.5 +2.0 لو weight ≤ target
muscle_gain (new_m - old_m) × 3.0 (new_f - old_f) × 0.8 +1.5 لو Strength + High Protein combo
general_fitness (new_f - old_f) × 2.0 (new_m - old_m) × 1.5 +1.5 لو fitness ≥ target
endurance (new_f - old_f) × 3.5 +1.0 لو Cardio Workout —

5.3 Safety Penalties (W₃ + W₄)

هذه تضمن إن الـ Agent ما يوصي بتمارين خطيرة:

# Progressive Fatigue Penalty — متصاعدة
if new_fatigue >= 4:  reward -= 4.0   # Severe → Injury Risk
elif new_fatigue >= 3: reward -= 2.0  # High → Unsustainable
elif new_fatigue >= 2 and old_fatigue < 2:
                       reward -= 0.5  # Early Warning

# Energy Depletion
if new_energy <= 0:   reward -= 2.5   # Exhaustion
elif new_energy <= 1:  reward -= 0.8  # Low Energy

🛡️ Value Alignment — من Lecture 7

الـ Fatigue Penalty (-4.0) هي أقوى عقوبة في النظام بأكمله. هذا بالتصميم — نريد الـ Agent يتعلم أن Safety أهم من Progress. في RL Theory، هذا يسمى Value Alignment: التأكد أن الـ Objective Function بتاع الـ Agent متوافقة مع مصلحة المستخدم.

5.4 Diversity Bonus

# عقوبة لو كرر نفس الـ Action يومين متتاليين
if self._last_action == (workout_key, nutrition_key):
    reward -= 0.3

هذا يمنع الـ Agent من الوقوع في Loop — مثلاً يكرر "HIIT + Deficit" كل يوم. التنويع مهم لمنع الـ Overuse Injuries.

5.5 Potential-Based Reward Shaping (PBRS)

من RL Literature (Ng et al. 1999): الـ PBRS يعطي الـ Agent Dense Reward كل Step بدل ما ينتظر نهاية الـ Episode:

φ(s) = -state_distance_to_goal(s)   # Potential Function
shaping = γ · φ(s') - φ(s)          # PBRS Formula
reward += shaping × 0.5             # Scale Factor

💡 ليش PBRS ما يغير الـ Optimal Policy؟

Ng et al. أثبتوا رياضياً أن هذا الشكل المحدد F = γ·φ(s') - φ(s) هو الشكل الوحيد اللي يسرّع التعلم بدون ما يغير الـ Optimal Policy. أي شكل ثاني (مثل F = φ(s') بس) ممكن يخلي الـ Agent يتعلم Policy خاطئة.

7.1 الـ Accumulator Mode لـ A* Search

لما A* يبني Search Tree، كل Node لازم يحمل نسخة من الـ Accumulators بتاعته:

8. Heuristic Function — state_distance_to_goal()

من Lecture 3: الـ A* Search يحتاج Heuristic Function h(n) لتقدير المسافة من الـ Current State للـ Goal. لازم تكون Admissible (ما تبالغ).

def state_distance_to_goal(self, state):
    fitness, energy, weight, muscle, fatigue, _ = state
    distance = 0

    if self.goal_type == 'weight_loss':
        distance += max(0, weight - target_weight) × 2       # Weight Gap
        distance += max(0, target_fitness - fitness) × 1.5    # Fitness Gap
    elif self.goal_type == 'muscle_gain':
        distance += max(0, target_muscle - muscle) × 2
        distance += max(0, target_fitness - fitness) × 1.5
    # ...
    distance += fatigue × 0.5    # Fatigue Penalty
    return distance

✅ Admissibility Proof

الـ Heuristic Admissible لأن: الـ Coefficients (×2, ×1.5) أقل من الـ Actual Cost بتاع كل Step. مثلاً، نزول مستوى Weight واحد يحتاج ~20 يوم من Caloric Deficit (تكلفة حقيقية ≈ 40+)، بينما الـ Heuristic تقدّرها بـ 2 فقط. هذا دائماً Underestimate → Admissible.

هذه الدالة ترشّح الـ Workouts حسب الـ Equipment والـ Time:

def get_available_workout_indices(equipment='gym', time_available=60):
    valid = []
    for i, workout in enumerate(WORKOUT_DATABASE):
        # لازم يكون عنده المعدات الكافية
        if workout_level <= user_level:
            # ولازم عنده وقت كافي
            if workout.duration_minutes <= time_available:
                valid.append(i)
    return valid if valid else [0]  # دائماً Rest متاح

💡 ليش return [0] كـ Fallback؟

لو المستخدم ما عنده أي معدات وعنده 10 دقائق فقط — ما فيه تمرين يناسبه. بدل ما يعطي Error، نرجع [0] (Rest) لأن الراحة دائماً متاحة بدون معدات وبدون وقت.
